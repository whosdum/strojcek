# Per-Phone Booking Rate Limit — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a per-phone rate limit of 3 online bookings per rolling 24h window to `createBooking`, preventing SMS/email/slot abuse from a single phone number.

**Architecture:** Add a second `prisma.appointment.count()` guard inside `createBooking` (server action), run before the existing global 30/h check. Relocate both rate-limit constants into `src/lib/constants.ts`. Add a composite DB index on `(customer_phone, created_at)` so the count query stays fast. Update `build` script to run `prisma migrate deploy` on every deploy.

**Tech Stack:** Next.js 16 (server actions), Prisma 7 with PrismaPg adapter, PostgreSQL, TypeScript, date-fns (`subHours`).

**Source spec:** [docs/superpowers/specs/2026-04-24-per-phone-booking-limit-design.md](../specs/2026-04-24-per-phone-booking-limit-design.md)

**Note on testing:** The spec explicitly keeps automated tests out of scope (no test framework in project today). Each task ends with a **manual verification** step the engineer runs from the dev server plus targeted DB/CLI checks. Commits happen in logical units, not after every step.

## Task 2: Add per-phone rate-limit check to `createBooking`

**Files:**
- Modify: `src/server/actions/booking.ts:27-39` (insert new check between phone normalization and existing global check)

Add the new guard immediately after `normalizePhone(data.phone)` and **before** the existing global-count check. This ordering rejects a single-phone bot on the cheaper per-phone query first; the global check still catches distributed bursts afterward.

- [ ] **Step 1: Insert the per-phone check**

In [src/server/actions/booking.ts](../../../src/server/actions/booking.ts), find this block starting around line 27:

```ts
    const data = bookingInputSchema.parse(input);
    const phone = normalizePhone(data.phone);

    // Global rate limit: max 30 bookings per hour
    const oneHourAgo = subHours(new Date(), 1);
    const recentCount = await prisma.appointment.count({
      where: { createdAt: { gte: oneHourAgo }, source: "online" },
    });
    if (recentCount >= GLOBAL_BOOKING_LIMIT) {
      return {
        success: false,
        error: "Príliš veľa rezervácií v krátkom čase. Skúste to o chvíľu.",
      };
    }
```

Replace it with:

```ts
    const data = bookingInputSchema.parse(input);
    const phone = normalizePhone(data.phone);

    // Per-phone rate limit: max 3 online bookings per rolling 24h window.
    // All statuses count (incl. CANCELLED/NO_SHOW) to prevent book→cancel→rebook abuse.
    const twentyFourHoursAgo = subHours(new Date(), 24);
    const phoneCount = await prisma.appointment.count({
      where: {
        customerPhone: phone,
        source: "online",
        createdAt: { gte: twentyFourHoursAgo },
      },
    });
    if (phoneCount >= PHONE_BOOKING_LIMIT_24H) {
      return {
        success: false,
        error:
          "Dosiahli ste maximálny počet rezervácií za 24 hodín. Pre ďalšiu rezerváciu zavolajte na 0944 932 871.",
      };
    }

    // Global rate limit: max 30 bookings per hour
    const oneHourAgo = subHours(new Date(), 1);
    const recentCount = await prisma.appointment.count({
      where: { createdAt: { gte: oneHourAgo }, source: "online" },
    });
    if (recentCount >= GLOBAL_BOOKING_LIMIT) {
      return {
        success: false,
        error: "Príliš veľa rezervácií v krátkom čase. Skúste to o chvíľu.",
      };
    }
```

The `subHours` import already exists at line 13 — no import changes needed.

## Task 3: Add composite DB index on `(customer_phone, created_at)`

**Files:**
- Modify: `prisma/schema.prisma` (add one line inside `model Appointment`)
- Create: `prisma/migrations/<timestamp>_add_phone_createdat_index/migration.sql` (generated)

Without an index, the new `prisma.appointment.count()` in Task 2 does a sequential scan over `appointments`. Fast today (small table), slow and potentially DoS-able as the table grows.

- [ ] **Step 1: Add the index declaration to the schema**

In [prisma/schema.prisma](../../../prisma/schema.prisma), find the `Appointment` model's index block (around line 221):

```prisma
  @@index([barberId, startTime])
  @@index([status, startTime])
  @@index([barberId, status, startTime])
  @@index([customerId])
  @@map("appointments")
```

Add the new index between the existing `@@index` lines and `@@map` (order: group all `@@index`, then `@@map`):

```prisma
  @@index([barberId, startTime])
  @@index([status, startTime])
  @@index([barberId, status, startTime])
  @@index([customerId])
  @@index([customerPhone, createdAt])
  @@map("appointments")
```

- [ ] **Step 2: Generate the migration**

Run:
```bash
npx prisma migrate dev --name add_phone_createdat_index
```

Expected output includes something like:
```
Applying migration `<timestamp>_add_phone_createdat_index`

The following migration(s) have been applied:

migrations/
  └─ <timestamp>_add_phone_createdat_index/
    └─ migration.sql

Your database is now in sync with your schema.
```

Inspect the generated `migration.sql` — it should contain a single statement:

```sql
CREATE INDEX "appointments_customer_phone_created_at_idx" ON "appointments"("customer_phone", "created_at");
```

If the SQL contains any additional changes (drops, alters, other renames), **stop** and investigate before committing — that would indicate the local DB has drifted from the schema.

- [ ] **Step 3: Verify Prisma client regenerated cleanly**

The `migrate dev` command also regenerates the Prisma client into `src/generated/prisma`. Run:

```bash
npx tsc --noEmit
```

Expected: exit code 0.

- [ ] **Step 4: Verify the index exists in Postgres**

Connect to the local Postgres and confirm:

```bash
npx prisma db execute --stdin <<< "SELECT indexname FROM pg_indexes WHERE tablename = 'appointments' AND indexname = 'appointments_customer_phone_created_at_idx';"
```

Expected: one row returned with that index name. (If `prisma db execute --stdin` is not available in your Prisma version, connect via `psql $DATABASE_URL` and run the same query.)

- [ ] **Step 5: Quick explain-check that the count query uses the index**

With a handful of rows in `appointments`, Postgres may still pick a seq scan because the table is too small — that's expected. On a larger table it will use the index. You can optionally force it:

```bash
npx prisma db execute --stdin <<< "EXPLAIN SELECT COUNT(*) FROM appointments WHERE customer_phone = '+421900000000' AND source = 'online' AND created_at >= NOW() - INTERVAL '24 hours';"
```

Skip this step if the output is confusing — the index existence check in step 4 is the load-bearing verification.

- [ ] **Step 6: Commit**

```bash
git add prisma/schema.prisma prisma/migrations/
git commit -m "feat(db): add index on appointments(customer_phone, created_at)"
```

## Self-review checklist (run after all tasks complete)

## Rollback

If a problem surfaces after deploy:

- **Logic regression:** revert the commits from Task 1, Task 2, Task 4 (the refactor, the feature, and the build-script change). The DB index from Task 3 can stay in place harmlessly.
- **Index causes unexpected write degradation** (highly unlikely for a composite index on two columns): create a follow-up migration with `DROP INDEX appointments_customer_phone_created_at_idx;`.
- **Build fails on deploy because migrate deploy errors:** Task 4's change is one line in `package.json`; revert it and run migrations manually via `npx prisma migrate deploy` on the production host until the underlying issue is fixed.
